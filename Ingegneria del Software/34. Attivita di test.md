# 34 Attivita di test

### Continous Testing & TDD

- Continous Testing -> testare di continuo e non solo alla fine
- TDD (Test Driven Development) -> scrivere i test prima del codice

### Automazione vs Umano

- Automazione -> indispensabile per velocita
- Umano -> è insostituibile per cose come usabilita o test esplorativi (UX)

## Unit Testing

livello piu basso

- Cosa testi -> singola unita di codice (metodo, classe, funzione)
- Come fai -> se l'unita necessita di altre dipendenze usi finti oggetti chiamati _Stub_ o _Driver_
- Vantaggio -> trovi errore subito e sai dov è

## Integration Testing

Metti insieme le unita e vedi se funzionano.

Strategie:
- Big Bang -> testi tutto alla fine (rischioso se esplode qualcosa)
- Top-Down -> parti dall alto e scendi (interfaccia, business, core)
- Bottom-Up -> il contrario
- Sandwich -> mix tra top down e bottom up

## System Testing

- Cosa testi -> l intero ambiente simulando uso reale end to end
- Obiettivo -> verificare sia req funzionali sia non funzionali

## Acceptance testing

momento della verità:
- chi lo fa -> cliente o utente finale
- obiettivo -> decidere se pagare la fattura e accettare il software

Casi comuni:
- __Alpha/Beta test__ -> versioni preliminari
- __Shadow Test__ -> il nuovo sistema gira insieme al vecchio per vedere se fornisce lo stesso output

## Testing Funzionale/Non Funzionale

### Funzionale (Black Box)

- domanda -> il software fa quello che ce scritto nei requisiti?
- metodo -> black box (no codice) guardi input e output

### Non Funzionale (Qualita)

- domanda -> Come si comporta il software?
- cosa guardi ->
    - performance
    - ux
    - sicurezza
    - affidabilita

## Testing di Sistema e Accettazione

tecniche per accettazione finale:
- __benchmark test__ test standard per prestazioni con il mercato
- __competitor test__ confronti il tuo software con quello dei competitor
