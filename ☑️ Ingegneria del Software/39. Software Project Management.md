# 39. Software Project Management

## 4 Pilastri del progetto

1. Outcome -> risultato finale
2. Work -> lavoro da fare
3. Schedule -> il tempo
4. Resources -> Le persone

Regola d'oro -> se tocchi uno si muovono tutti

Esempio: il cliente vuole il software in meno tempo, quindi devi aumentare le Resources o ridurre il Work

## Fasi dinamiche (ciclo di vita)

1. Conception -> idea iniziale -> Conviene farlo? (Costi benefici)
2. Definition -> si decide cosa fare. Qui nasce il __SPMP__ (piano di progetto)
3. Start -> si formano i team e si preparano gli strumenti (Kickoff)
4. Steady State -> fase operativa dove si scrive il codice
5. Termination -> consegna, chiusura e analisi post mortem (quindi cosa abbiamo imparato)

## Divide et Impera: la WBS

Concetto chiave -> __Scomposizione__

- WBS (Work Breakdown Structure) -> albero gerarchico che spezza il progetto in tanti piccoli pezzetti
- Task vs Attivita:
    - Task -> unita minima -> un responsabile e una durata
    - Attivita -> insieme di task
- Critical Path -> sequenza di task che determina durata minima progetto. se uno solo di questi tarda, a cascata ridarda l intero progetto.
    - Slack Time -> tempo in cui un task puo ritardare senza impattare la fine del progetto. nei critical paths lo slack time = 0 

## Strumenti di gestione

Si tratta di strumenti _burocratici_:

1. SPMP (Software Project Management Plan): bibbia del progetto. Definisce chi fa cosa, come e quando. Segue lo standard _IEEE 1058_. Diviso in 3 categorie:
    1. Piani Manageriali -> riguardano soldi tempi e rischi
    2. Piani Tecnici -> riguardano come si fa il software
        - modello -> agile, waterfall
        - strumenti -> linguaggio
        - infrastruttura
    3. Piani di supporto -> riguardano la qualita e l ordine
        - Configuration Manager -> git, versionamento, documentazione, qualitu assurance (QA)
2. Skill Matrix -> tabella che mappa chi sa fare cosa. utile per comprendere a chi assegnare cosa.

## DEliverables e Internal Work Products

- deliverables -> cio che consegni al cliente, manuale e software
- internal work products -> quello che serve al team ma che il cliente non vede (UML, casi di test, verbali riunioni interne)

## Come schedulare il task

ci sono 4 metodi standard:
- ASAP -> as soon as possible, default
- ALAP -> as late as possible, finisci piu tardi che puoi, utile per task meno importanti
- MSO -> must start on, deve iniziare esattamente in una data specifica
- MFO -> must finish on, deve terminare esattamente il ...
