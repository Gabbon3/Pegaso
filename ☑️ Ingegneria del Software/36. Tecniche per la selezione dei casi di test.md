# 36 Tecniche per la selezione dei casi di test

## Esplosione Combinatoria

se un metodo ha 3 parametri e ogni parametro ha 10 valori possibili ottieni gia 1000 combinazioni (10 * 10 * 10)

__Soluzione__: vanno usate tecniche di selezione sistematica. no scelte a random, scegli casi che hanno alta probabilita di trovare bug

## Concetti chiave

- Random vs Sistematico:
    - Random -> facile e veloce ma non hai controllo
    - Sistematico -> hai controllo e tracciabilita, + difficile e lungo

## Black e White box (ripasso)

black = solo esterno input e outpout
white = guardi anche il codice, hai visibilita su tutto

## Equivalence Class Testing (ECT)

se x accetta numeri da 1 a 100 ne scelgo 1 solo e mi vale per tutti.

### 1 Classi valide e non

- classi valide -> dati che il sistema _deve accettare_
- classi non valide -> dati che il sistema deve _rifiutare_
- __regola__: bisogna sempre prevedere un test per le classi valide e non

### 2 Debole vs Forte

- weak (debole) -> testi solo classi valide-valide. piu semplice ma rischi di non trovare combinazioni strane.
- strong (forte) -> testi tutte le combinazioni: valida-valida, valida-invalida...

### 3 Boundary Value Analysis (BVA)

- idea -> i programmatori sbagliano quasi sempre sulle cazzate (i confini), ad esempio < invece di <=
- tecnica -> invece di prendere numeri a caso prendi i _bordi_ (quindi nell esempio di prima con 1 <= x <= 100 prenderei 1 o 100 e 0 e 101)

## Category Partition Method (CPM)

ECT è per singoli campi, mentre il CPM è per interi sistemi.

in breve è un metodo strutturato per non tralasciare nulla:

si compone di 4 step principali:
1. analisi -> legge requisiti e trova parametri
2. partizionamento -> suddividi ogni parametro in categorie (es: password -> "corta" "lunga"...)
3. __vincoli__ -> usi regole per escludere combinazioni impossibili:
    - se password vuota non testi anche se contiene caratteri speciali...
4. generazione -> generi i test case combinando cio che ti è rimasto.

## Conclusioni

Testare tutto è impossibile, testare bene è d'obbligo.
l'obiettivo è fare test in maniera __intelligente__
