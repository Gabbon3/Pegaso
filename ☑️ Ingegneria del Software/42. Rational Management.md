# 42 Rationale Management

Risponde alla domanda: perche questa cosa è stata fatta cosi?

## Cos'è

- definizione -> disciplina che si occupa di catturare e gestire le motivazioni dietro certe scelte progettuali
- problema -> il rationale management documenta il percorso mentale (perche ho scelto A e non B)
- obiettivo -> creare una _memoria razionale_. se sai perche una cosa è stata fatta in un modo hai controllo su di essa e puoi cambiarla in futuro

## Processo Ingegneristico

Serve in ogni fase:
- requisiti -> perche abbiamo escluso quella feature
- architettura -> perche abbiamo usato questo DB e non un altro?
- codice -> perche ho usato questo pattern strano?

## Rischio della perdita di conoscenza

- Problema del Turnover -> se il senior che ha scritto il codice se ne va, il Rationale rimane con lui, la conoscenza è persa per sempre
- risultato -> chi rimane ha paura di toccare il codice perche non sa se cera un motivo valido per una determinata stranezza

## Approcci e Sfide

- Sfida -> la resistenza culturale, i dev odiano scrivere documentazione perche lo si vede come tempo perso
- Vantaggio -> accellera l onboarding, se il rationale fosse scritto sarebbe piu facile comprendere il perche subito.

## Cos è il Design Rationale

5 Elementi:
1. Decisione presa -> la scelta finale
2. Alternative considerate -> cosa è stato valutato o scartato
3. Criteri decisionali -> le metriche usate
4. Contesto -> il cliente voleva....
5. Motivazioni -> ragionamento finale

## Concetto Issue-Based

il design non è un processo lineare ma una discussione
- si parte da un problema __issue__
- si propongono soluzioni
- si litiga (argomentazioni, dibattiti) finche non si decide
- questi modelli servono a tracciare questo "dibattito costruttivo"

## IBIS: Issue Based Information System

Serve per i cosiddetti __Wicked Problems__ -> problemi serva risoluzione ovvia

struttura a 3 nodi:

1. Issue I -> la domanda o il problema (che db usiamo?)
2. Position P -> la proposta di soluzione (usiamo oracle, usiamo postgres)
3. Argument A -> pro o contro di una posizione (Oracle costa troppo, postgres è gratis)

## QOC: Questions, Options, Criteria

Evoluzione ingegneristica di IBIS

1. Questions -> problema
2. Options -> le alternative
3. Criteria -> metri di giudizio

Differenza con IBIS: QOC enfatizza i criteri. Costringe a dire perche ho scelto A

## DLR: Design Relationale Language

Si usa solo in contesti specifici come aerei, ospedali, banche dove i rischi sono alti.

è estremamente dettagliato, traccia tutto, anche i Goal (Obiettivi, è molto piu costoso e formale).
Non si usa nei metodi AGILI

## Integrazione

Questo processo di motivazione va INTEGRATO negli strumenti che usato ogni giorno i dev, ad esempio nei commenti sul codice, nei ticket, nei commit di git...
